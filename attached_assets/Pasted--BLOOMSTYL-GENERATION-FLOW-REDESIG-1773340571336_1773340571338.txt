═══════════════════════════════════════════════════════════════
CRITICAL BUG FIX — GENERATION LOOP (DO THIS FIRST)
═══════════════════════════════════════════════════════════════

The generate button triggers a loop and freezes.
Fix before touching any UI or feature work.

LIKELY CAUSES TO INVESTIGATE:
1. Async function called without await — promise never 
   resolves, retry logic fires repeatedly
2. State mutation inside render cycle — state change 
   triggers re-render which triggers generation again
3. Missing error boundary on AI call — failed fetch 
   silently retries in a loop
4. Event listener attached multiple times — button click 
   fires N times because listener added on every render

FIX REQUIREMENTS:

Add a generation lock:
  let isGenerating = false
  
  async function handleGenerate() {
    if (isGenerating) return  // hard stop on double-fire
    isGenerating = true
    try {
      await generateWorksheet()
    } catch (err) {
      console.error('Generation failed:', err)
      showErrorState(err.message)
    } finally {
      isGenerating = false  // always release lock
    }
  }

Wrap ALL AI calls in try/catch/finally.
Never let a failed fetch leave isGenerating = true.

Add a generation timeout:
  const controller = new AbortController()
  const timeout = setTimeout(() => controller.abort(), 30000)
  
  fetch(apiUrl, { signal: controller.signal, ...options })
    .finally(() => clearTimeout(timeout))

If generation takes > 30 seconds, abort and show:
  "Taking longer than expected. [Try Again] [Simplify Request]"

Remove all retry-on-failure auto-logic.
Failed = show error state. Teacher clicks retry manually.

Audit all useEffect / event listeners.
Every listener must have a cleanup function.
No listener should be added inside a render loop.

PART 2 — PARALLEL JOB ARCHITECTURE
═══════════════════════════════════════════════════════════════

The core performance problem: everything is sequential.
AI call finishes → layout renders → customization unlocks.
This makes everything feel slow and blocked.

Replace with a parallel job pipeline where multiple 
things happen simultaneously.

── THE JOB QUEUE ─────────────────────────────────────────────

When teacher clicks "Create Worksheet," fire these 
jobs in parallel immediately:

JOB A — LAYOUT SKELETON (fires instantly, 0ms delay)
  Do not wait for AI.
  Immediately render 3 layout skeleton cards on screen.
  These are shimmer placeholder previews showing the
  structural shape of 3 different layout types.
  The teacher sees something appear before the AI responds.
  Layout skeleton renders in < 100ms.

JOB B — AI CONTENT GENERATION (fires immediately)
  Call Claude with the topic + subject + grade.
  Request structured JSON: title, passage, questions, vocab.
  Stream the response.
  As each field arrives, fill it into the skeleton cards.
  Title appears first (arrives fast).
  Then directions. Then passage. Then questions.
  Each field fades in as it arrives.

JOB C — LAYOUT VARIATION ANALYSIS (fires immediately, 
  separate from Job B)
  Separate lightweight AI call (or local logic):
  Determine the 3 best layout patterns for this content type.
  Returns in < 2 seconds typically.
  Skeleton cards update to show actual layout structure 
  when this arrives.

JOB D — CUSTOMIZATION PANEL POPULATION (fires immediately)
  While Jobs B and C are running, the customization panel
  on the right side of the screen is already visible and
  usable. It is pre-populated with smart defaults based
  on the subject and grade selected.
  Teacher can start adjusting font, colors, layout 
  preferences NOW — before generation completes.
  Changes queue up and apply when the canvas renders.

JOB E — ASSET PRELOADING (fires immediately, lowest priority)
  Begin loading fonts, clipart suggestions, and shape 
  assets relevant to the topic in the background.
  These will be ready by the time teacher opens the editor.

── JOB COORDINATOR ───────────────────────────────────────────

Implement a simple job coordinator:

  const jobs = {
    skeleton:       { status: 'pending', result: null },
    content:        { status: 'pending', result: null },
    layouts:        { status: 'pending', result: null },
    customization:  { status: 'pending', result: null },
    assets:         { status: 'pending', result: null }
  }

  function updateJob(name, status, result = null) {
    jobs[name] = { status, result }
    reconcileCanvas()  // called after every job update
  }

  function reconcileCanvas() {
    // Rules for what to show based on job states:
    
    // Skeleton always shows immediately
    if (jobs.skeleton.status === 'complete') {
      renderSkeletonCards(jobs.skeleton.result)
    }
    
    // Fill content into existing skeletons as it arrives
    if (jobs.content.status === 'streaming') {
      fillStreamingContent(jobs.content.result)
    }
    
    // Update layout shapes when layout analysis arrives
    if (jobs.layouts.status === 'complete') {
      updateSkeletonStructures(jobs.layouts.result)
    }
    
    // Customization panel always available immediately
    if (jobs.customization.status === 'complete') {
      activateCustomizationPanel(jobs.customization.result)
    }
  }

── STREAMING CONTENT FILL ────────────────────────────────────

Use streaming API responses (stream: true).
Do not wait for full response before showing anything.

Request Claude to output JSON in a predictable field order:
  1. title (arrives in first ~100 tokens)
  2. grade_level
  3. directions
  4. vocabulary (array)
  5. passage (array of paragraphs)
  6. questions (array)
  7. layout_suggestion

Parse the stream incrementally:
  - When title field closes → render title in all 3 cards
  - When directions closes → render directions
  - When each vocabulary word arrives → add to vocab list
  - When each paragraph arrives → add to passage
  - When each question arrives → add to question list

Use a streaming JSON parser (or simple buffer approach):
  Accumulate tokens in a buffer.
  On each token, attempt to extract any newly completed 
  JSON field using regex or incremental JSON parse.
  Apply completed fields to canvas immediately.

Visual effect:
  Each section of the worksheet fades in 
  as its content arrives (150ms fade, ease-in).
  The worksheet literally builds itself in front of 
  the teacher in real time.
  This makes a 4-second AI response feel like 
  < 1 second because something is always happening.

PART 5 — TRANSITIONS AND ANIMATION SPEC
═══════════════════════════════════════════════════════════════

All current transitions are laggy because they are likely:
  - Triggering full re-renders instead of CSS transitions
  - Using setTimeout instead of requestAnimationFrame
  - Not using GPU-accelerated properties
  - Missing will-change hints

RULES FOR EVERY TRANSITION:

USE ONLY GPU-ACCELERATED CSS PROPERTIES:
  ✓ transform (translate, scale, rotate)
  ✓ opacity
  ✗ Never animate: width, height, top, left, margin, 
    padding, border-width, background-color directly
    (use transform: scale() instead of width changes)
    (use opacity instead of visibility/display)

ALL DURATIONS:
  Instant feedback (button press, tap): 0–50ms
  Element appearance (fade in): 150ms
  Panel slide: 180ms
  Card selection expand: 220ms
  Page-level transitions: 280ms max
  Nothing should ever take > 300ms to start responding

EASING:
  Elements entering: ease-out (starts fast, decelerates)
  Elements leaving: ease-in (starts slow, accelerates out)
  Elastic/bounce effects: cubic-bezier(0.34, 1.2, 0.64, 1)
    — use only for card selection, nothing else

SUBJECT BUTTON SELECT:
  transform: scale(1.03)
  transition: transform 80ms ease-out, 
              background-color 80ms ease-out
  No layout shift. No reflow.

STEP 2 REVEAL (topic input appearing):
  Initial state: opacity: 0, transform: translateY(-8px)
  Final state: opacity: 1, transform: translateY(0)
  Transition: 180ms ease-out
  Trigger: add class 'visible' via JS — CSS does the rest

CARD SKELETON APPEARANCE:
  Cards render at opacity: 0 immediately on generate click.
  Stagger their appearance: card 1 at 0ms, 2 at 40ms, 3 at 80ms
  Each fades from opacity 0 → 1 over 200ms
  Cards are in the DOM instantly. CSS handles the fade.

CONTENT FILL-IN:
  Each content section (title, directions, questions) has:
  initial: opacity: 0.3 (shimmer state)
  filled: opacity: 1, transition: opacity 150ms ease-in
  No layout shift when content arrives.
  Reserve space for content using min-height on containers.

CARD → EDITOR TRANSITION:
  Selected card gets class 'expanding'
  CSS: transform: scale(1) → scale(20) over 220ms
  Simultaneously: editor canvas fades in at opacity 0 → 1
  At transition midpoint: swap card DOM for canvas DOM
  Result: seamless visual continuity, no white flash

NO TRANSITIONS ON:
  Hover color changes (just use CSS :hover, no JS)
  Toggle switches (immediate)
  Text input changes (immediate)
  Dropdown changes (immediate)

WILL-CHANGE DECLARATIONS:
  Add will-change: transform, opacity to:
    .layout-card
    .subject-button
    .panel-slide
    .skeleton-card
  Remove will-change after transition completes to 
  free GPU memory.

PART 6 — CANVAS PERFORMANCE
═══════════════════════════════════════════════════════════════

VIRTUALIZE THE CANVAS:
  Only render canvas elements that are in the viewport.
  Off-screen elements are tracked in data but not in DOM.
  On scroll or pan, swap elements in/out of DOM.

DEBOUNCE EXPENSIVE OPERATIONS:
  Canvas re-render on element drag: 
    Use requestAnimationFrame, not setInterval or setTimeout
  
  Right panel property updates on element select:
    Debounce 16ms (one frame)
  
  Auto-save:
    Debounce 2000ms after last change

AVOID LAYOUT THRASH:
  Never read and write DOM properties in the same frame.
  Batch all DOM reads together, then all writes.
  Use documentFragment for multi-element inserts.

IMAGE LOADING:
  Lazy load all images (loading="lazy" or IntersectionObserver)
  Pre-generate low-res placeholder (4px blur-up technique)
  WebP format for all raster images
  Cache all images in browser Cache API

FONT LOADING:
  All Google Fonts preloaded in <head> with 
  <link rel="preload" as="font">
  Use font-display: swap — never block render on fonts
  Subset fonts to characters needed (A-Z, a-z, 0-9, punctuation)
